# 合规即代码 — 用《立意》构建可审计的意图链

🌍: 简体中文（普通话） / [English](compliance-as-code.en.md)

**读者：** 技术负责人、合规工程师、QA 主管、外包项目经理——需要向审计方或客户证明"代码确实实现了需求"的人。开发者请先阅读 [README](../README.zh.md)。

> ⚠️ **本文档是设计阶段的叙事草案，不是产品承诺。** 《立意》的协议格式与 CLI 工具仍在活跃演变中，预期会有破坏性变更。如果您是合规工程师或项目经理，请暂时忽略本文档——当前版本面向的是项目贡献者和技术评估者，用于验证设计方向是否正确。待工具稳定后，本文档会更新为正式的场景指南。

## 问题

当审计员问"这段代码确实实现了监管要求吗？"时，你的回答是什么？

大多数团队的现状：

- 需求写在 Jira/禅道/飞书多维表格里，代码在另一头。两者之间的对应关系靠人脑记忆。
- AI 智能体写了代码，审阅者点了 Approve，但没人能证明代码的*意图*与需求*一致*。
- 代码改了，需求文档没跟上。三个月后再审计，没人知道哪些变更影响了哪个合规条款。
- 外包团队离场后，业务逻辑散落在代码注释、即时消息和不断过期的文档中。

这不是开发效率问题，是**可问责性**问题。

---

## 《立意》如何解决

《立意》不是"AI 写代码更快"的工具。它是一套**意图基础设施**——让每一行代码都能追溯到它应当满足的意图或需求，并在二者失去同步时，由 CI 工具自动报警。

### 第一步：需求锚定到代码

在代码中，用 `@liyi:requirement` 标记监管条款或业务要求。需求文本就在标记旁边，与代码共存于同一仓库、同一版本管理之下。

```python
# @liyi:requirement aml-transaction-monitoring
# 每笔交易必须在 T+0 内完成反洗钱初筛。
# 超过阈值金额的交易必须触发人工审核流程。
# 筛查结果必须保留至少五年。
```

不是另起一份文档。不是填进外部系统。需求*就在代码旁边*。

### 第二步：代码-意图绑定

AI 智能体为每个代码条目推断意图——该函数*应当*做什么——并记录在 `.liyi.jsonc` sidecar 文件中，同时声明该条目与哪些需求相关：

```jsonc
{
  "item": "screen_transaction",
  "intent": "对交易执行反洗钱初筛：验证金额阈值，超限触发人工审核队列，记录筛查结果并附时间戳。",
  "related": { "aml-transaction-monitoring": null },
  "source_span": [42, 78]
}
```

有人审阅意图后标记 `"reviewed": true`，即完成人类确认。五行意图描述而非五十行实现代码——审阅负担降低一个数量级。

### 第三步：变更检测

当代码或需求发生变更时，`liyi check` 自动检测：

```plain
screen_transaction: ⚠ REQ CHANGED — requirement "aml-transaction-monitoring" updated
validate_amount:    ⚠ STALE — source changed since spec was written
log_screening:      ✓ reviewed, current
```

- **需求变了** → 所有声明了与该需求相关的代码条目被传递标记。开发者必须逐一确认：代码还满足更新后的需求吗？
- **代码变了** → 意图被标记为过时。审阅者检查：变更是否偏离了声明的意图？
- **两者同步** → 绿灯通过。

这就是*可审计性*的含义：不是一份滞后的合规文档，而是一个**活的、可验证的链条**——从需求到意图到代码实现，每个环节变更可追踪、漂移可检测。

## 为什么这对合规审计很重要

| 审计需求 | 传统方式 | 《立意》方式 |
| -------- | ------- | ----------- |
| 需求到代码的追溯 | 手动维护映射表，易过时 | `@liyi:requirement` + `@liyi:related` 构成机器可验证的双向链接 |
| 变更影响分析 | 全局搜索、人脑记忆 | `liyi check` 自动标记所有受影响代码条目 |
| 审阅证据 | PR 的 Approve 按钮（仅证明有人点了） | `"reviewed": true` + `source_hash` = 特定版本的意图已经特定人类确认 |
| 知识留存 | 依赖原团队成员的记忆 | `.liyi.jsonc` 持久存在于仓库中，与代码共同版本管理 |
| 漂移检测 | 定期人工审计（季度/年度） | 每次推送自动检查，在 CI 中运行 |

### 与工单系统的区别

Jira（或禅道、Linear 等）管理的是*工作流*——谁做、什么时候做、做到哪了。《立意》管理的是*意图*——这段代码应当做什么、它与哪个需求相关、二者是否还同步。

当代码改了但 Jira 里的需求描述没改，Jira 不会报警。`liyi check` 会。

两者互补而非替代。Jira 在项目管理层面是对的工具；《立意》在代码-需求一致性层面补上了 Jira 覆盖不到的缺口。

### 溯源

`.liyi.jsonc` 与源码共同纳入版本管理。审阅行为（将 `"reviewed": true` 写入 sidecar）也是提交历史的一部分——VCS 自动记录谁在何时批准了哪条意图。`.liyi.jsonc` 不是配置文件，而是一份随代码演进的审计日志。

## 场景

### 金融合规

银行反洗钱规则调整。业务部门更新了需求文本中的阈值和审核流程描述。

```plain
$ liyi check
screen_transaction:   ⚠ REQ CHANGED — requirement "aml-transaction-monitoring" updated
escalate_suspicious:  ⚠ REQ CHANGED — requirement "aml-transaction-monitoring" updated
log_screening:        ✓ current
generate_report:      ✓ current
```

两个函数被标记。开发者打开每个函数，对照新需求检查实现，确认或修改后重新标记为已复核。审计时，可直接查看该需求名称对应的所有条目的复核历史（`git log` 即可追溯）。

### 外包项目交付

客户在项目中途改了需求。项目经理更新了代码中的 `@liyi:requirement` 文本。

```plain
$ liyi check
calculate_discount:   ⚠ REQ CHANGED — requirement "discount-rules-v2" updated
apply_coupon:         ⚠ REQ CHANGED — requirement "discount-rules-v2" updated
render_price:         ✓ current
```

受影响的代码被自动定位——不是全局搜索，不是靠开发者回忆。每个标记出来的函数被逐一处理并标记为已复核后，`.liyi.jsonc` 就是交付物的一部分：证明"你改的需求，我们都处理了"。

### 遗留系统治理

外包团队要走了。在离场前，AI 智能体读取现有代码，为每个函数推断意图并记录在 `.liyi.jsonc` 中。这是*知识沉淀*——不是完美的文档，而是"这段代码大概是做什么的"的结构化记录，接手的内部团队不必从零开始理解业务逻辑。

后续日常中，每次代码变更 `liyi check` 都在 CI 中运行。意图漂移被持续检测，知识不再随人员流动而消散。

## 技术特性

| 特性 | 合规价值 |
| ---- | ------- |
| 零网络依赖 | 可在离线内网运行，数据不出境 |
| 零 LLM 依赖 | 检查结果 100% 可复现——同一代码、同一 sidecar，永远产生相同结论 |
| 语言无关 | 支持遗留系统（Java、PHP）与现代系统（Rust、Go、Python）的混合架构 |
| 轻量部署 | 单个二进制文件，无运行时依赖，无需采购审批 |
| CI 友好 | 标准退出码（0 = 通过，1 = 检查失败，2 = 内部错误），可直接接入现有流水线 |
| 渐进式采用 | 从一个文件开始试点，逐步推广到全仓库，无需一次性改造 |

**为什么合规检查需要确定性工具？** 如果检查工具本身调用概率性 LLM，今天检查通过，明天同一代码可能因模型行为差异而失败——这在审计视角下是不可接受的。`liyi check` 基于哈希比对和行号追踪，是数学确定性的，100% 可复现。意图的*推断*由 AI 智能体完成，但*检查*不依赖 AI。

## 快速开始

```bash
# 安装
cargo install --path crates/liyi-cli

# 在代码中添加需求标记和 @liyi:related 注解
# 由 AI 智能体生成 .liyi.jsonc 意图规格

# 填充哈希并检查
liyi check --fix --root .

# 集成到 CI
liyi check --root .
```

详细用法参见 [README.zh.md](../README.zh.md)。

《立意》不是又一套需要采购预算的 SaaS 平台。它是一个文件约定加一个命令行工具——可以今天在一个项目试用，明天推广到全团队，无需审批流程。
